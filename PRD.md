# PRD: Прости, Извини

## 1. Введение

**Прости, Извини** — локальное веб-приложение-дневник для осознанного учёта случаев, когда пользователь говорит «прости» или «извини»: кому, в какой ситуации и какие выводы он для себя сделал.

**Ценность:** помочь заметить частоту и контекст извинений и поддержать рефлексию, а не только считать слова.

**Целевая аудитория:** люди, которые хотят лучше понимать свои привычки в общении, работают над границами или осознанностью в извинениях.

---

## 2. Цели и метрики успеха

### Продуктовые цели

- Пользователь может **быстро зафиксировать** факт извинения с контекстом и краткой рефлексией.
- Пользователь видит **динамику во времени** (день, неделя, месяц) и **график частоты**.
- Данные **остаются на устройстве**, без обязательной регистрации.

### Метрики успеха (намерения для v1)

- Регулярность ввода: осмысленное число записей в неделю (без серверной аналитики точные пороги не фиксируются).
- Использование экранов статистики: пользователь хотя бы периодически открывает сводку за неделю/месяц.

---

## 3. Объём MVP (in scope)

| Область | Содержание |
|--------|------------|
| **Запись** | Дата и время события (по умолчанию «сейчас», с возможностью правки). Тип фразы: **«прости»**, **«извини»** или **оба / не различаю** (одно поле выбора). Поля: **кому**, **почему** (контекст), **анализ / выводы** (что понял, что сделает иначе). |
| **Список** | Просмотр записей с группировкой по дням. Фильтр периода: **день**, **неделя**, **месяц** (календарные границы). |
| **Счётчики** | Количество извинений за выбранный **день**; агрегаты за выбранную **неделю** и **месяц** (общее число записей). Отдельно — разбивка по типу фразы («прости» / «извини» / оба) в сводках. |
| **Визуализация** | График частоты по **дням** (неделя или месяц). **Две основные серии:** число записей с типом «прости» и с типом «извини» по каждому дню. Записи с типом **«оба / не различаю»** не дублируются в этих сериях; они учитываются в **общем** счётчике за период и отображаются **отдельной серией** «оба» (или эквивалентная подпись в легенде). |
| **Хранение** | Только **локально в браузере**, предпочтительно **IndexedDB**. Регистрации и облака нет. |
| **Резервная копия** | **Экспорт данных в JSON** (скачивание файла) в составе MVP — чтобы снизить риск потери при очистке профиля браузера. Импорт из JSON — желательно в той же версии или в следующем минорном релизе; в MVP достаточно экспорта, импорт — в backlog, если не успевает. |

### Решения по продукту (зафиксировано)

- **Несколько записей в день:** разрешены; «сколько раз в день» = число записей с датой в этот календарный день.
- **«Прости» и «извини»:** в сводках — **раздельные** счётчики для выбранных типов; **общий** счётчик — по всем записям за период. На графике — серии «прости» и «извини» плюс серия «оба» для неразделённых записей.
- **Экспорт JSON:** входит в **MVP** как мера защиты данных на устройстве.

---

## 4. Вне объёма MVP (out of scope)

- Облачное хранение, аккаунты, синхронизация между устройствами.
- Push-уведомления и напоминания (backlog).
- Социальные функции, публичный шаринг записей.
- PWA-установка и офлайн-манифест (опционально после MVP).
- Мультиязычный интерфейс (в MVP — **русский**).

---

## 5. Пользовательские сценарии (user stories)

1. Как пользователь, я хочу **за пару шагов добавить запись** после разговора, чтобы не откладывать рефлексию.
2. Как пользователь, я хочу видеть **сколько раз сегодня** я зафиксировал извинения и **разбивку по «прости» / «извини»**, чтобы понимать привычку формулировок.
3. Как пользователь, я хочу **листать записи по дням** и фильтровать неделю/месяц, чтобы вспоминать контекст.
4. Как пользователь, я хочу видеть **график за неделю или месяц**, чтобы замечать тренды, а не отдельные дни.
5. Как пользователь, я хочу **скачать резервную копию в JSON**, чтобы не потерять дневник при смене браузера или очистке данных (с пониманием, что восстановление через импорт может появиться следующим шагом).

---

## 6. Функциональные требования

1. **Создание записи:** форма с полями: дата-время, тип фразы, кому, почему, анализ/выводы. Валидация: обязательны минимум контекст или рефлексия по политике продукта — **минимум одно текстовое поле непустое** (например «почему» или «анализ»).
2. **Редактирование и удаление** существующей записи.
3. **Список записей** с сортировкой от новых к старым, группировка по календарному дню.
4. **Фильтр периода:** день / неделя / месяц с понятными границами (локальное время пользователя).
5. **Агрегаты:** суммарное число записей за период; счётчики по типу фразы где применимо.
6. **График:** отображение ежедневных частот за выбранный диапазон (неделя или месяц); пустые дни — ноль или отсутствие столбца по согласованию с библиотекой графиков.
7. **Пустые состояния:** при отсутствии записей — подсказка создать первую запись.
8. **Экспорт:** выгрузка всех записей в один JSON-файл с метаданными версии схемы (рекомендуется поле `schemaVersion`).

---

## 7. Нефункциональные требования

- **Производительность:** интерфейс остаётся отзывчивым при **сотнях** записей (ленивая подгрузка списка — по желанию).
- **Конфиденциальность:** данные не передаются на серверы продукта; в интерфейсе — краткое **предупреждение**, что очистка сайта в браузере удалит данные, и рекомендация делать экспорт.
- **Доступность:** базовая поддержка клавиатуры и контраста в рамках возможностей Ant Design.
- **Совместимость:** современные браузеры с поддержкой IndexedDB.

## 8. Модель данных (черновик)

Сущность **ApologyEntry** (имя может быть локализовано в коде как `apology` / `entry`):

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | string (UUID) | Уникальный идентификатор |
| `createdAt` | string (ISO 8601) | Момент события (для сортировки и графиков) |
| `phraseType` | `'prosti' \| 'izvini' \| 'both'` | Тип фразы; `both` — когда не различает или оба варианта |
| `toWhom` | string | Кому (свободный текст) |
| `reason` | string | Почему / контекст |
| `reflection` | string | Анализ / выводы |

Теги в **MVP не обязательны**; при расширении — опциональный массив `tags: string[]`.

Экспортируемый JSON-файл: массив записей и/или объект `{ schemaVersion, exportedAt, entries }`.

## 9. UI/UX принципы

- Интерфейс на **русском языке**.
- Короткие формы, понятные подписи полей.
- **Ant Design** — основная библиотека компонентов (формы, списки, выбор даты, модальные окна).
- Графики: отдельная библиотека по выбору команды (**@ant-design/charts**, Recharts или аналог) — требование: читаемая легенда и подписи осей по датам.

---

## 10. Риски и допущения

- **Потеря данных** при смене браузера, переустановке ОС или очистке хранилища — частично смягчается **экспортом JSON**; импорт в MVP не обязателен, но снижает трение при миграции.
- Нет единого «облачного» бэкапа — осознанное ограничение v1.
- Точность графиков завязана на **локальный часовой пояс** браузера.

---

## 11. Технический контекст (реализация)

- **Стек:** Vite, React, TypeScript; UI — Ant Design; состояние — **Zustand**; персистенция — **IndexedDB** (через слой репозитория или middleware), без обязательного бэкенда в v1.
- **Точка входа:** [src/App.tsx](src/App.tsx) — эволюция в основной экран приложения и при необходимости маршрутизацию.

## 13. План задач для агента (цикл как в ralph)

Реализацию MVP удобно вести **итеративно**: агент читает PRD, берёт следующую задачу из [`tasks.json`](tasks.json), выполняет её, затем **обновляет статус** в `tasks.json` и при необходимости краткую заметку в поле `notes`.

### Принципы

1. **Источник правды по статусам** — файл [`tasks.json`](tasks.json) в корне репозитория (не править историю задач внутри PRD вместо JSON).
2. **Один активный шаг:** не больше одной задачи со статусом `in_progress` одновременно.
3. **Порядок:** по возрастанию `order` (или `id`), если не указана блокировка; при `blocked` — сначала снять причину блокировки в `notes` или выполнить зависимости.
4. **После работы:** выставить `done` или вернуть в `pending`, если задача не завершена; при ошибке/ожидании пользователя — `blocked` и причина в `notes`.
5. **Связь с PRD:** у каждой задачи есть `prdRefs` — номера разделов или пунктов этого документа для проверки критериев приёмки.

Допустимые значения `status`: `pending` | `in_progress` | `done` | `blocked`.

### Сводка задач (дублирует `tasks.json` для чтения человеком)

| ID | Кратко | PRD |
|----|--------|-----|
| T1 | Типы и слой IndexedDB (CRUD записей) | §8, §3 хранение |
| T2 | Zustand + загрузка/сохранение в IndexedDB | §11 |
| T3 | Каркас UI: layout, русская локаль Ant Design | §9 |
| T4 | Форма создания/редактирования записи + валидация | §6 п.1–2, §3 запись |
| T5 | Список записей, группировка по дням, фильтр периода | §6 п.3–4, §3 список |
| T6 | Сводки и счётчики (день/неделя/месяц, по типам фразы) | §3 счётчики, §6 п.5 |
| T7 | График частоты по дням (неделя/месяц, серии) | §3 визуализация, §6 п.6 |
| T8 | Экспорт JSON (`schemaVersion`) + предупреждение о локальных данных | §6 п.8, §7 конфиденциальность |
| T9 | Пустые состояния, полировка UX | §6 п.7 |

Полный перечень полей и актуальные статусы — только в [`tasks.json`](tasks.json).
